0. The shared setup (read this first — it applies to every model)

Before the individual models, here is the machinery they all sit inside. Every model is just a different "brain" plugged into the same data pipeline.

Data. NASA POWER daily weather for each city, 2021‑01‑01 → 2025‑12‑12, with forward forecasting from 2026‑01‑01. Raw channels include temperature (TMP2m), relative humidity (RH2m), mean‑sea‑level pressure (PRmsl), wind components (U10m,V10m), specific humidity (QV2m), etc.

Feature engineering (features.py → engineer_features). From those raw channels the code builds a rich daily feature table: lagged values (1,2,3,7,14,30 days), rolling means/std/min/max (windows 3,7,14,30), pressure/temperature tendencies and differences, anomalies (value vs trailing mean), wind speed/ direction (and their sin/cos), and seasonality (doy_sin/cos, month_sin/cos, week_sin/cos, is_monsoon/summer/winter/premonsoon). Crucially, every engineered feature uses .shift(1) so the row for day D only ever uses information available up to day D‑1. This prevents target leakage.

Two feature modes (features.py → select_features).

  • Multivariate (MV): all engineered features — 77 columns. The model sees the whole weather state (temperature, humidity, pressure, wind, season…).
  • Univariate (UV): only the target's own derived features + seasonality. Counts differ slightly per target: TMP2m = 30, RH2m = 27, PRmsl = 29. The model only sees the history of the thing it is predicting, plus the calendar.

Three targets. TARGETS = ["TMP2m","RH2m","PRmsl"], units °C, %, Pa. One model is trained per (city, mode, model‑type, target). So for a single city+mode there are 6 model‑types × 3 targets = 18 models (this is the "18 models" you see in the tuning status line).

Windowing (engine.py → build_sequences). Features and target are scaled with MinMaxScaler (to [0,1]). Then a sliding window of SEQ_LEN = 14 days is formed:

X[i] = scaled_features[i : i+14]     # shape (14, F)   — 14 days of features
y[i] = scaled_target[i+14]           # shape (1,)      — the NEXT day's target

So the universal task is: given the last 14 days of features, predict the next day's value of one target (scaled), which is then inverse‑transformed back to real units.

Training (engine.py → train_model). Loss = MSE; optimizer = Adam (lr=1e-3, weight_decay=1e-5); up to 120 epochs with early stopping and ReduceLROnPlateau; first 80% of history for training, the tail held out for validation/backtest. In the live daily cycle the model is fine‑tuned ~6 epochs or fully retrained every 7th verified day / 1st of month.

Quantum stack (used only by QLSTM, QGRU, HQNN). PennyLane default.qubit state‑vector simulator, interface="torch", diff_method="backprop". The variational circuit is AngleEmbedding (X‑rotations, data → angles) followed by StronglyEntanglingLayers (depth q_depth=2), measured as the Pauli‑Z expectation ⟨Z⟩ on each wire (a real number in [‑1,1] per qubit).

Where models are defined. All six classes live in src/models.py, registered in _REGISTRY, and instantiated through one factory: build_model(name, input_size, hparams, mode). The factory reads the per‑mode size table POC_CONFIG and applies any tuned overrides (hidden, q_depth, n_qubits). It also stamps _iwf_hp/_iwf_mode onto the model so the exact architecture is saved with the weights.

Model 1: LSTM (Long Short‑Term Memory)

CLASSICAL
Mode Multivariate and Univariate (separate instances)Target TMP2m, RH2m, or PRmsl (one model each)Type Classical (recurrent neural network)
Next‑day forecast of one weather target from a 14‑day window.

Full architecture

Input x of shape (batch, 14, F) — 14 days, F features per day (F = 77 MV / 30‑ish UV). Layers, in order: 1. nn.LSTM(input_size=F, hidden_size=H, batch_first=True) — H = 32 (MV) / 24 (UV). 2. Take the hidden state at the last timestep: lstm(x)[0][:, -1, :] → (batch, H). 3. nn.Linear(H, 1) → the scaled next‑day prediction (batch, 1).

No dropout, no normalization, single LSTM layer (matches the POC). Activation non‑linearities live inside the LSTM cell (sigmoid gates + tanh).

Layer‑by‑layer breakdown

  • LSTM layer. Reads the 14 daily feature vectors one at a time, maintaining a cell state (long‑term memory) and hidden state (short‑term memory). At each step three gates decide what to forget, what new information to store, and what to output. Input: the 14‑step sequence. Output: a hidden vector per step; we keep only the last one, which is a learned summary of the whole window.
  • Linear head. Maps the H‑dimensional summary to a single number — the scaled forecast. Inverse‑scaling (sc_y) turns it back into °C / % / Pa.

Internal working

For each day t in the window, the LSTM updates: forget gate f_t = σ(W_f·[h_{t-1},x_t]), input gate i_t, candidate g_t=tanh(...), cell c_t = f_t⊙c_{t-1} + i_t⊙g_t, output gate o_t, hidden h_t = o_t⊙tanh(c_t). The cell state c_t is the key: it can carry information across many days with little decay, which is why LSTMs handle longer dependencies than plain RNNs. After 14 steps, h_14 summarizes the trajectory; the Linear head reads it to predict tomorrow.

Mathematical / conceptual intuition

Weather is autocorrelated and seasonal — tomorrow looks a lot like a smooth continuation of recent days plus seasonal drift. The LSTM's gated memory is good at "remember the trend, react to recent changes." Strengths: captures temporal momentum and regime persistence; robust default. Weaknesses: more parameters and slower than GRU; can over‑smooth sharp transitions; needs enough data.

Project implementation notes

  • Class LSTMModel in src/models.py; registry key "lstm".
  • Built via build_model("lstm", F, hp, mode); POC_CONFIG sets hidden 32/24.
  • Tunable override: hidden (Optuna can change hidden size). dropout is accepted but the POC architecture ignores it.
  • Param counts: 14,241 (MV) / 5,401 (UV).

Input/output behavior

  • Input: 14×F window of scaled engineered features.
  • Output: one scaled number → inverse‑scaled to the target's unit = the next day's TMP2m/RH2m/PRmsl.

Strengths / weaknesses

+ Strong on long temporal dependencies and seasonality; reliable baseline. − Heaviest classical recurrent option here; slower; can blur abrupt changes.

Model 2: QLSTM (Quantum‑enhanced LSTM)

HYBRID
Mode MV and UVTarget TMP2m / RH2m / PRmslType Hybrid (classical LSTM + variational quantum circuit)
Same next‑day forecast, but blends a quantum feature branch with

Full architecture

Two parallel branches over the same input (batch,14,F): 1. Classical branch: nn.LSTM(F, H) (H = 32 MV / 24 UV) → last hidden (batch,H). 2. Quantum branch (on the last day only, x[:,-1,:]):

  • nn.Linear(F → n_qubits) then tanh(...)·π → angles in (‑π,π).
  • VQC: AngleEmbedding(X) → StronglyEntanglingLayers(depth 2) → ⟨Z⟩ per wire → (batch, n_qubits). (n_qubits = 6 MV / 4 UV.)
  • nn.Linear(n_qubits → c_hidden) → ReLU (c_hidden = 16 MV / 12 UV).

3. Fusion: concatenate [lstm_last (H), q_out (c_hidden)] → nn.Linear(H + c_hidden → 1) = scaled forecast.

Layer‑by‑layer breakdown

  • LSTM branch — identical role to Model 1: a temporal summary of the 14 days.
  • Linear→tanh·π (encoder) — squeezes F features of the last day into n_qubits rotation angles, bounded to (‑π,π) so they are valid qubit rotations.
  • VQC — the quantum layer; see the quantum section below. Returns one expectation value per qubit.
  • Linear+ReLU — lifts the n_qubits quantum readout into a small classical feature vector.
  • Fusion Linear — learns how much to trust the recurrent summary vs the quantum features when producing the final number.

Internal working

The LSTM processes the full window (recurrence as in Model 1). In parallel, the last day's feature vector is encoded into qubit rotations, transformed by the entangling circuit, and measured. The two representations are concatenated and a linear layer fuses them. Gradients flow through both branches together (diff_method="backprop" differentiates the simulated circuit too), so the quantum weights train jointly with the network.

Mathematical / conceptual intuition

The hypothesis: a small quantum circuit can build non‑linear, entangled combinations of the latest features that a linear/ReLU layer would need many more parameters to express. If those features carry signal, the fusion layer will use them. Strengths: adds an expressive non‑classical feature map on top of a proven LSTM. Weaknesses: the quantum branch only sees the last day (not the sequence), the simulator is slow, and at 6 qubits the extra capacity is tiny — gains are not guaranteed.

Quantum explanation

  • Qubits (6 MV / 4 UV): each wire is one quantum degree of freedom; together they form a 2^n‑dimensional state the circuit manipulates.
  • Encoding: classical data → Linear(F→n_q) → tanh·π → AngleEmbedding with rotation="X", i.e. apply RX(angle_i) to wire i. Data becomes rotation angles (this is angle encoding).
  • Variational layers: StronglyEntanglingLayers, depth 2 — each layer applies a trainable 3‑angle rotation (Rot) to every qubit and then a ring of CNOT entanglers, so information mixes across qubits. Trainable tensor shape (depth, n_qubits, 3) (= 36 weights for 6 qubits, 24 for 4 qubits).
  • Measurement: ⟨Z⟩ on each wire → n_qubits real numbers in [‑1,1].
  • Back to classical: those values go through Linear→ReLU and are concatenated with the LSTM output.
  • Why this design: angle encoding + strongly‑entangling layers is the standard, hardware‑agnostic VQC; backprop on default.qubit makes it trainable end‑to‑end on a simulator.

Project implementation notes

  • Class QLSTMModel; VQC built by _make_vqc(input_size, n_qubits, q_depth).
  • Quantum params: proj Linear(F→n_q) + circuit weights. MV ≈ 504, UV ≈ 148.
  • Tunable: hidden, n_qubits, q_depth. Total params 14,873 (MV)/5,621 (UV).

Input/output behavior

Same as LSTM: 14×F window in → one inverse‑scaled next‑day target out. The quantum branch additionally consumes the last day's features.

Strengths / weaknesses

+ Keeps the LSTM's temporal strength; adds an entangled non‑linear feature map. − Quantum branch ignores the sequence; slowest to train; benefit uncertain at small qubit counts.

Model 3: GRU (Gated Recurrent Unit)

CLASSICAL
Mode MV and UV · Target: TMP2m / RH2m / PRmsl · Type: Classical (recurrent)
A lighter recurrent alternative to the LSTM for the same next‑day task.

Full architecture

1. nn.GRU(F, H, batch_first=True) — H = 24 (MV) / 16 (UV). 2. Last‑timestep hidden gru(x)[0][:, -1, :] → (batch, H). 3. nn.Linear(H, 1) → scaled forecast.

Layer‑by‑layer breakdown

  • GRU layer. Like the LSTM but with two gates (reset + update) and no separate cell state — the hidden state carries memory directly. Fewer parameters, often comparable accuracy.
  • Linear head. Maps the last hidden state to the prediction.

Internal working

Per day: update gate z_t = σ(...), reset gate r_t = σ(...), candidate ñ_t = tanh(W·[r_t⊙h_{t-1}, x_t]), new hidden h_t = (1‑z_t)⊙h_{t-1} + z_t⊙ñ_t. The update gate interpolates between "keep old memory" and "take new info," giving GRUs efficient control of memory with fewer weights than an LSTM.

Mathematical / conceptual intuition

Same autocorrelation/seasonality rationale as the LSTM, but a leaner model — often the better choice when data is limited or speed matters. Strengths: fast, fewer params, strong baseline. Weaknesses: slightly less capacity for very long dependencies than an LSTM.

Project implementation notes

  • Class GRUModel, key "gru"; hidden 24/16 from POC_CONFIG.
  • Tunable hidden. Params 7,441 (MV) / 2,321 (UV) — the lightest classical model.

Input/output behavior

14×F window in → one inverse‑scaled next‑day target out.

Strengths / weaknesses

+ Efficient, fast, solid accuracy; great default for smaller data. − A touch less expressive than LSTM on long‑range patterns.

Model 4: QGRU (Quantum‑enhanced GRU)

CLASSICAL
Mode MV and UV · Target: TMP2m / RH2m / PRmsl · Type: Hybrid (GRU + VQC)
The GRU counterpart of QLSTM — a leaner hybrid.

Full architecture

Mirrors QLSTM with a GRU instead of an LSTM: 1. GRU branch: nn.GRU(F, H) (H = 24 MV / 16 UV) → last hidden. 2. Quantum branch (last day): Linear(F→n_q)→tanh·π→VQC(AngleEmbedding+ StronglyEntanglingLayers)→⟨Z⟩ → Linear(n_q→c_hidden)→ReLU (n_qubits 6/4, c_hidden 12/8). 3. Fusion: concat [gru_last(H), q_out(c_hidden)] → Linear(H+c_hidden→1).

Layer‑by‑layer breakdown / internal working

Identical logic to QLSTM (see Model 2) with the GRU's two‑gate recurrence in place of the LSTM cell. The GRU summarizes the window; the VQC adds an entangled non‑linear map of the latest day; a linear layer fuses them.

Quantum explanation

Same VQC recipe as QLSTM (angle X‑encoding + strongly‑entangling depth‑2, ⟨Z⟩ readout). Quantum params ≈ 504 (MV) / 148 (UV). The smaller c_hidden (12/8) makes QGRU the lightest hybrid.

Project implementation notes

  • Class QGRUModel, key "qgru"; VQC via _make_vqc. Params 8,041 (MV)/2,517 (UV).
  • Tunable hidden, n_qubits, q_depth.

Input/output behavior

14×F window in → next‑day target out (inverse‑scaled).

Strengths / weaknesses

+ Cheapest hybrid; keeps GRU efficiency while testing quantum features. − Same caveats as QLSTM: quantum branch sees only the last day; simulator cost.

Model 5: ANN (feed‑forward Artificial Neural Network / MLP)

CLASSICAL
Mode MV and UV · Target: TMP2m / RH2m / PRmsl · Type: Classical (non‑recurrent)
A non‑sequential baseline — does temporal memory actually help?

Full architecture

Uses only the last day of the window: x[:, -1, :] → (batch, F). Then a multilayer perceptron:

  • MV: Linear(F→128)→ReLU→Linear(128→64)→ReLU→Linear(64→32)→ReLU→Linear(32→1).
  • UV: Linear(F→64)→ReLU→Linear(64→32)→ReLU→Linear(32→16)→ReLU→Linear(16→1).

Layer‑by‑layer breakdown

Each Linear+ReLU block learns non‑linear combinations of the previous layer's features, progressively compressing F inputs down toward the single prediction. No recurrence, no memory across days — only the most recent day's engineered features (which already include lags/rollings, so some history leaks in through the features themselves).

Internal working

A straight forward pass: last‑day features → stacked dense layers → one number. There is no hidden state to update; "time" only enters through the lag/rolling features computed in features.py.

Mathematical / conceptual intuition

Tests the hypothesis that engineered lag/rolling features already capture the temporal signal, so a plain MLP on the latest row can compete with recurrent models. Strengths: simple, fast, often surprisingly strong because the features are rich; the most parameters in MV (20,353) thanks to the wide 128‑unit layer. Weaknesses: no genuine sequence modeling; relies entirely on feature engineering; can overfit the widest layer.

Project implementation notes

  • Class ANNModel, key "ann"; hidden_units (128,64,32)/(64,32,16).
  • Note it ignores all but x[:,-1,:]. Params 20,353 (MV) / 4,609 (UV).

Input/output behavior

Takes the 14×F window but reads only day 14; outputs the inverse‑scaled next‑day target.

Strengths / weaknesses

+ Simple, fast, strong when features are well engineered; good control baseline. − No temporal recurrence; widest layer is parameter‑heavy in MV.

Model 6: HQNN (Hybrid Quantum Neural Network)

CLASSICAL
Mode MV and UV · Target: TMP2m / RH2m / PRmsl · Type: Hybrid (dual‑branch quantum + classical MLP)
The "most quantum" design — a quantum branch and a classical branch

Full architecture

Operates on the last day x[:,-1,:] (F features). Two encoders: 1. Quantum‑encoder branch: MLP Linear(F→cu)→GELU…→Linear(→n_qubits) (cu = (64,32) MV / (32,16) UV), then tanh·π, then VQC. Inside the VQC there is another Linear(n_q→n_q)→tanh·π→AngleEmbedding(X)→StronglyEntanglingLayers→⟨Z⟩. Output: (batch, n_qubits) (n_qubits = 6, both modes). 2. Classical‑encoder branch: MLP Linear(F→cu)→GELU… → (batch, cu[-1]) (cu[-1] = 32 MV / 16 UV). 3. Head: concat [q_out (n_q), c_out (cu[-1])] → Linear(→16)→GELU→Linear(16→1).

Layer‑by‑layer breakdown

  • Quantum encoder MLP — compresses F features (with GELU activations) down to n_qubits numbers suitable for angle encoding.
  • VQC — the quantum core; produces n_qubits expectation values.
  • Classical encoder MLP — a parallel purely‑classical embedding of the same last‑day features.
  • Fusion head — a tiny 2‑layer MLP that learns to combine quantum and classical embeddings into the final prediction.

Internal working

Both branches read the last day. The quantum branch maps features → angles → entangled state → measurements; the classical branch maps features → dense embedding. Their outputs are concatenated and the head emits one scaled value. All weights (including the simulated circuit) train together via backprop.

Mathematical / conceptual intuition

This is the cleanest test of "quantum vs classical on equal footing": two symmetric encoders feeding one head. If the quantum branch contributes useful, hard‑to‑classically‑replicate features, the head will weight it. Strengths: most explicitly hybrid; smallest quantum parameter count (78) yet a full classical companion. Weaknesses: no recurrence at all (last‑day only, like the ANN); the double encoding (encoder MLP and the VQC's internal projection) is somewhat redundant; simulator cost.

Quantum explanation

  • Qubits: 6 (both modes). Encoding is two‑stage: an MLP brings F→6, then the VQC's own Linear(6→6)→tanh·π→AngleEmbedding(X) turns those into RX rotations.
  • Variational layers: StronglyEntanglingLayers depth 2 (weights (2,6,3)=36).
  • Measurement: ⟨Z⟩ on 6 wires → 6 numbers.
  • Quantum params: VQC proj Linear(6→6) (42) + circuit weights (36) = 78.
  • Why: keeps the quantum register fixed at 6 qubits regardless of mode, so the circuit is a stable, comparable "quantum feature extractor" feeding the head.

Project implementation notes

  • Class HQNNModel, key "hqnn"; n_qubits=6, classical_hidden (64,32)/(32,16). Params 15,061 (MV) / 3,605 (UV); quantum part 78 both modes.
  • Tunable n_qubits, q_depth.

Input/output behavior

14×F window in (reads day 14 only) → inverse‑scaled next‑day target out.

Strengths / weaknesses

+ Most genuinely hybrid; symmetric quantum/classical comparison; compact quantum core. − No sequence modeling; double‑encoding redundancy; simulator cost.

  • Purely classical: LSTM, GRU, ANN. Hybrid quantum‑classical: QLSTM, QGRU, HQNN.
  • Parameter differences: ANN is widest in MV; GRU is lightest overall; hybrids add a small VQC on top of their classical core (so each hybrid ≳ its classical twin).

  • Performance tradeoffs (expected): recurrent models (LSTM/GRU) usually lead on smooth, autocorrelated targets like temperature; ANN is a strong feature‑driven baseline; hybrids are the research question — they may match or edge out their classical twins on some targets/modes, at much higher training cost.
  • Interpretability: plain MLP (ANN) and recurrent gates are reasonably interpretable; the VQC inside the hybrids is effectively a black box.

10. Simple explanation (as if teaching someone new to ML / quantum ML)

Imagine predicting tomorrow's temperature from the last two weeks of weather.

  • GRU / LSTM are like a person who reads the diary of the last 14 days in order, keeping a running mental note ("memory") of the trend, then guesses tomorrow. LSTM has a bigger notebook; GRU is a faster note‑taker.
  • ANN is like a person who only glances at yesterday's summary card (which already lists recent averages and changes) and guesses — no reading the diary day by day.
  • Quantum models (QLSTM, QGRU, HQNN) add a tiny "quantum gadget." You feed a few numbers into it as rotation angles; the gadget spins and tangles them together in a way ordinary arithmetic can't easily copy, then you read a few numbers back out. Those readings become extra clues the model can use. QLSTM/QGRU bolt this gadget onto the diary‑reader; HQNN builds a quantum reader and a normal reader side by side and combines them.

The whole project asks one research question: do the quantum gadgets actually help, and at what cost? Everything else (data, features, 14‑day window, scaling, training) is kept identical so the comparison is fair.

11. How to present each model in a report / viva

LSTM — "A single‑layer LSTM over a 14‑day window; the last hidden state feeds a linear head. Gated cell memory captures seasonal momentum. Our strongest classical recurrent baseline." Be ready to: sketch the gate equations; explain why we take only the last timestep.

QLSTM — "LSTM in parallel with a variational quantum circuit on the latest day; outputs are concatenated and fused. Angle‑encoded RX + strongly‑entangling layers, ⟨Z⟩ readout, trained by backprop on a simulator." Be ready to: justify why the quantum branch is only last‑day; state the quantum param count (504 MV).

GRU — "A leaner recurrent model (two gates, no cell state); fewest parameters; fast. Often matches LSTM on limited data." Be ready to: contrast GRU vs LSTM gates.

QGRU — "The GRU analogue of QLSTM; lightest hybrid (smaller fusion width). Same VQC recipe." Be ready to: explain it isolates whether a cheaper recurrent core still benefits from the quantum branch.

ANN — "A feed‑forward MLP on the last day only; no recurrence. Tests whether our engineered lag/rolling features already encode the temporal signal." Be ready to: explain why it can still be competitive (rich features), and that it ignores days 1–13.

HQNN — "A symmetric dual‑branch hybrid: a quantum encoder (MLP→VQC) and a classical encoder (MLP), fused by a small head. Fixed 6‑qubit register; smallest quantum footprint (78 params)." Be ready to: explain the two‑stage encoding and why it is the fairest quantum‑vs‑classical comparison.

General viva tips. Emphasize the controlled setup: identical data, 77/30 features, 14‑day window, MinMax scaling, MSE/Adam, per‑(city,mode,target) models. Be honest that quantum simulation is the cost driver and that at 6 qubits the quantum advantage is a hypothesis being tested, not a guarantee. Know your numbers: SEQ_LEN=14, targets in °C/%/Pa, 18 models per city+mode, quantum = AngleEmbedding(X) + StronglyEntanglingLayers(depth 2) + ⟨Z⟩.
